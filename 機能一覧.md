# Sullivan 機能一覧とロジック詳細

本ドキュメントは、Sullivan の**現行実装**と、プロダクトの意図（AI伴走 × 人の支援）を対応づけて整理したものです。

## 1. 目指す価値と設計方針

Sullivan は、次の価値仮説を前提に設計されています。

- AI が「個別最適化タスク（出題・採点・復習提案）」を担う
- 人が「情意面・継続支援（見守り・声かけ・習慣化）」に集中する
- その分業により、低価格でも高密度な支援を成立させる

## 2. ユーザーロール

- `ADMIN`: 全体管理（ユーザー/教室/カリキュラム/問題）
- `HEAD_TEACHER`: 講師機能 + 講師ユーザー作成
- `TEACHER`: 担当教室の生徒管理、指導記録、印刷支援
- `STUDENT`: 学習、履歴、復習、印刷
- `MATERIAL_AUTHOR`: 教材作成・編集（問題/Revision 中心）
- `PARENT`: スキーマ上は定義あり（現行UIは限定的）

## 3. 機能一覧

### 3.1 認証・認可

- Supabase Auth によるログイン/ログアウト
- ロールは `user.app_metadata.role` を使用
- 初回既定パスワード時は強制変更
- 教室スコープに基づくアクセス制御（講師は同教室生徒のみ）

### 3.2 学習サイクル（Print & Scan）

- 生徒/教科指定で問題を自動選出して印刷
- 問題セットを QR 化して解答用紙に埋め込み
- 解答用紙を Drive に投入すると採点キューへ連携

### 3.3 AI採点・フィードバック

- Gemini による採点（A/B/C/D）
- 日本語フィードバックを保存
- 画像はローカル OpenCV QR 解析（`scripts/qr_reader.py`）、失敗時は Gemini でフォールバック
- 採点結果は `LearningHistory` と学習状態テーブル（`UserProblemState` / `UserCoreProblemState`）に反映
- 問題本文は `ProblemRevision`（PUBLISHED）を真のソースとし、`Problem.answer` / `acceptedAnswers` は `publishedRevision` から派生する片方向同期

### 3.4 復習優先度と単元解放

- 問題単位: `UserProblemState.priority` と `lastAnsweredAt`
- 単元単位: `UserCoreProblemState.priority`, `isUnlocked`, `isLectureWatched`
- 単元解放判定（`src/lib/progression.ts`）:
  - 回答率 >= 40%（`UNLOCK_ANSWER_RATE`）
  - 正答率 >= 50%（`UNLOCK_CORRECT_RATE`）
- 条件達成で次の CoreProblem を自動解放

### 3.5 Print Gate（学習順序の担保）

- アンロック済みで講義動画未視聴の単元がある場合、印刷をブロック
- 単元フォーカス画面へ遷移して視聴を促す

### 3.6 人による伴走支援

- 講師が指導記録（面談/指導/その他）を登録
- 生徒のプロフィール・メモを管理
- 担当教室内で学習進捗を横断確認

### 3.7 AI家庭教師（PREMIUM教室）

- チャット相談（`/api/tutor-chat`）
- 音声相談（Gemini Live + `/ws`）
- 教室プランが `PREMIUM` の場合のみ利用可

### 3.8 非認知アンケート

- 90日間隔でアンケート表示
- 5カテゴリからランダム出題し、カテゴリスコアを保存
- カテゴリ:
  - `GRIT`
  - `SELF_EFFICACY`
  - `SELF_REGULATION`
  - `GROWTH_MINDSET`
  - `EMOTIONAL_REGULATION`

### 3.9 ゲーミフィケーション

- XP/レベル計算
- 連続学習日数（Streak）
- 実績（Achievements）
- Realtime イベントで UI 更新

### 3.10 通知・リアルタイム

- `realtime_events` テーブルにイベントを発行
- 主なイベント:
  - `grading_completed`
  - `gamification_update`
  - `core_problem_unlocked`

## 4. コアロジック（Sullivan Logic）

### 4.1 出題選定ロジック（`src/lib/print-algo.ts`）

1. `getReadyCoreProblemIds` で「アンロック済み + 講義視聴済み」単元を抽出
2. 候補問題を取得（通常印刷では「紐づく全 CoreProblem が ready」条件）
3. スコア計算（重みは `PRINT_CONFIG` で集中管理）
   - 未着手: `UNANSWERED_BASE * WEIGHT_UNANSWERED` をベース、`order` を僅かに優先（再現性）
   - 既着手: `経過日数 * FORGETTING_RATE * WEIGHT_TIME`（`TIME_SCORE_CAP` で上限クリップ）
   - 正解済: `CORRECT_PENALTY` を減算（再演を後ろに送る）
   - 不正解: `WEAKNESS_BONUS * WEIGHT_WEAKNESS` を加算（弱点優先）
4. 未着手枠を `NEW_QUOTA_RATIO` / `NEW_QUOTA_MIN` で保証した上でスコア上位から採用（デフォルト 30 問）

### 4.2 採点後の状態更新（`src/lib/grading-service.ts`）

- `LearningHistory` へ回答結果を保存
- `UserProblemState.priority` 更新
  - A: -10, B: -5, C: +5, D: +10
- `UserCoreProblemState.priority` 更新
  - 正解時: 関連 CoreProblem を減点
  - 不正解時: 推定弱点 CoreProblem を加点
- 解放判定と `core_problem_unlocked` イベント発行

### 4.3 ワーカー分離（運用）

- Webサービスは webhook 受信とジョブ発行に専念
- Workerサービスが重い採点処理を担当
- Cloud Tasks + Cloud Run IAM で保護されたジョブのみ処理

## 5. 意図メモとの対応（現時点）

- 「AIと人の分業」:
  - AI採点・出題最適化・AIチューター
  - 指導記録・講師ダッシュボード
- 「非認知能力の可視化」:
  - 90日アンケート（GRIT/自己効力感など）
- 「継続・習慣化」:
  - Streak、実績、Realtime通知

## 6. これからの検証観点（ドキュメント上の整理）

現行実装は PoC 検証に必要な主要基盤を備えています。今後の事業検証では、以下の指標整理が重要です。

- 学習密度: 学習時間、正答率、単元到達
- 継続指標: 継続率、提出率、再開率
- 情意指標: アンケートカテゴリ推移
- 運営指標: 講師工数、同時対応人数、単価と採算
