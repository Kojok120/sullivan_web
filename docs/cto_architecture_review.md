# CTOレビュー資料: Sullivanアーキテクチャ評価と改善提案

対象: CEO向け  
作成者: 新任CTO  
目的: 現状実装を踏まえたリスク整理と、事業成長に耐える改善方針の提案

## 1. 結論サマリー（経営判断の要点）
Sullivanは、Next.jsを中核に**AI採点パイプラインまで一体化したモノリシック構成**で、MVPとしては合理的です。  
一方で、**セキュリティ（鍵管理・Webhook認証）**と**スケーラビリティ（分散処理・リアルタイム通知・重いAI処理）**に構造的リスクがあり、ユーザー規模拡大時に「採点失敗」「遅延」「情報漏洩」の顕在化が懸念されます。

本資料では、以下の3点を優先改善領域として提案します。  
1) **秘密情報の管理強化とWebhook認証の強化**  
2) **採点処理の分離とジョブ管理の導入（冪等性・再試行）**  
3) **リアルタイム通知とロック制御の分散対応**

---

## 2. 現状アーキテクチャ（実装ベース）

### 2.1 コア構成
- **Webアプリ**: Next.js 16 (App Router, Server Actions)  
- **DB**: PostgreSQL + Prisma ORM  
- **認証**: Supabase Auth (SSR)  
- **AI採点**: Google Gemini API  
- **スキャン取り込み**: Google Drive API (Webhook検知)  
- **ジョブキュー**: Upstash QStash  
- **状態管理**: Upstash Redis (Drive Watch状態)  
- **QR解析**: Python + OpenCV（ローカル実行）  
- **通知**: SSE（サーバー内イベントエミッタ）

### 2.2 主要データフロー

**A. プリント生成**  
1. 出題アルゴリズムで問題選定  
2. QRコードに生徒IDと問題IDを埋め込み  
3. A4レイアウトで出力

**B. 採点処理**  
1. Google Drive Webhook受信  
2. Drive内の新規ファイル検出 → QStashにジョブ発行  
3. /tmpにダウンロード → QR解析  
4. Geminiで答案読取・採点  
5. DB更新（学習履歴・進捗・アンロック判定）  
6. SSEで通知

**C. 進捗・アンロック**  
回答率・正答率に基づき次の単元を解放

## 4. 主要課題とリスク（経営視点の影響）

| 優先度 | 課題 | 事業影響 | 技術的背景 |
| --- | --- | --- | --- |
| 高 | サービスアカウント鍵がリポジトリ内に存在 | 情報漏洩リスク / コンプラ事故 | `service-account.json` がルートに存在 |
| 高 | Webhook認証が弱い | 外部からの不正トリガー・過剰課金 | WebhookはChannel IDのみで検証 |
| 高 | 採点処理とWebアプリが同居 | タイムアウト・採点失敗・スケール不全 | API Route内で重い処理を実行 |
| 中 | ロック/デバウンスが単一インスタンス前提 | 複数台運用で二重採点や衝突 | `/tmp`ファイルとメモリ変数で制御 |
| 中 | SSE通知が単一プロセス依存 | リアルタイム通知が不安定 | EventEmitterは分散不可 |
| 中 | 冪等性/重複防止が弱い | 同一答案の二重採点 | fileId基準のジョブ管理が無い |
| 低 | ロール判定の不整合 | 認可挙動の不整合 | login側がuser_metadata参照 |
| 低 | Driveチェックが10件限定 | 大量投入時の取りこぼし | Paginationが未対応 |
| 低 | DBインデックス不足 | 速度低下 | `LearningHistory`等が無索引 |

## 5. 改善提案（優先度順）

### 5.1 短期（0-4週間）
1. **秘密情報の整理と鍵のローテーション**  
   - `service-account.json` を削除し、Secret Manager/環境変数で運用  
2. **Drive Webhookの署名/トークン検証**  
   - `x-goog-channel-token` を使用し、固定トークンで認証  
3. **ジョブ冪等性の導入**  
   - `GradingJob` 테ーブルを追加し `fileId` にUnique制約  
4. **分散ロック/デバウンスのRedis化**  
   - ロック状態をRedisで管理（TTL付き）  
5. **ログインロール参照の統一**  
   - `app_metadata` を唯一の権限ソースにする

### 5.2 中期（1-3ヶ月）
1. **採点処理を専用ワーカーに分離**  
   - Cloud Run等に移設し、Next.js本体はUIに集中  
2. **リアルタイム通知基盤の置き換え**  
   - Supabase Realtime / Ably / Pusher 等へ移行  
3. **Drive依存の低減**  
   - GCS/S3への直接アップロード導線を検討

### 5.3 長期（3-6ヶ月）
1. **監視/可観測性の強化**  
   - Sentry + 構造化ログ + メトリクス  
2. **データガバナンス・保持ポリシー策定**  
   - 学習ログ・画像データの保持期間定義  
3. **DB最適化と分析基盤整備**  
   - インデックス追加、集計基盤の整備
