# 中学校 理科・数学問題対応 仕様書

更新日: 2026-03-31 (JST)  
ステータス: ドラフト v1

本書は、Sullivan に中学校の理科・数学問題を追加するにあたり、問題仕様、問題管理方法、印刷方法、他科目との差分、採点への影響、技術的懸念点を実装前に整理するための仕様書である。

## 結論

- 現状の「`Problem.question` に文字列を保持し、そのまま印刷する」方式では、理科・数学で必要になる数式、図形、グラフ、表、観察図、作図余白を安定して扱えない。
- 理科・数学は、生テキストではなく「構造化ブロック + アセット参照 + 印刷設定 + 採点設定」で問題を管理する前提に切り替えるべきである。
- 印刷は「10問固定 + 1問1行の解答用紙」から、問題形式に応じたテンプレート型へ移行するべきである。
- 自動採点は問題形式ごとに適用範囲を明確に制限し、途中式、作図、記述考察は `manual_review` または `rubric_ai` を明示的に許容するべきである。

## 1. 背景と目的

この章で決めること: なぜ本仕様が必要か、何を決めるための文書かを明確にする。

現状の Sullivan は、英語の短答問題を中心に成立している。英語では、問題文と正答を文字列で管理し、印刷時もテキストをそのままレイアウトする構造で十分成立しやすい。一方で、理科・数学では次の要素が頻出する。

- 数式
- 図形
- グラフ
- 表
- 実験図
- 観察結果
- 作図用の余白
- 複数空欄や複数要素から成る解答

これらは、単なるテキストの改行だけでは表現しきれず、問題本文の表現、解答欄、印刷レイアウト、採点方式のすべてに影響する。

本仕様書の目的は、次を実装前に整理することである。

- 理科・数学の問題仕様
- 問題の管理方法
- 印刷方法
- 他科目との違い
- 技術的な懸念点

本書は実装手順書ではなく、将来の実装判断に必要な設計前提を固定するための文書である。

## 3. 現状整理

この章で決めること: 現在の実装が何を前提にしているかを、事実ベースで整理する。

現状実装から見ると、問題管理と印刷は英語の短答問題に最適化されている。

### 3.1 データ構造

- `Problem` は `question`, `answer`, `acceptedAnswers`, `grade`, `masterNumber`, `videoUrl` などを持つが、問題本文自体は `question: String` の文字列管理である。
- `CoreProblem` は単元ノードとして使われており、`Problem` は `CoreProblem[]` に紐づく。
- `Subject -> CoreProblem -> Problem` という構造は、現状の学習順序制御、講義動画、印刷導線、解放ロジックと既に結合している。

参照:

- [`../prisma/schema.prisma`](../prisma/schema.prisma)

### 3.2 管理画面

現状実装から見ると、問題作成・編集画面は次の入力を前提としている。

- 問題文: `Textarea`
- 解答: `Input`
- 学年: `Input`
- 動画 URL: `Input`
- 単元紐付け: `CoreProblemSelector`

つまり、本文中に図版、数式、表、複数の解答欄、図形用余白を持つ問題は、入力モデルとして想定されていない。

参照:

- [`../src/app/admin/problems/components/problem-dialog.tsx`](../src/app/admin/problems/components/problem-dialog.tsx)
- [`../src/lib/tsv-parser.ts`](../src/lib/tsv-parser.ts)

### 3.3 印刷

現状実装から見ると、印刷は次の前提で組まれている。

- 1セット 10 問固定
- 問題文は `question` テキストを HTML エスケープして表示
- 解答用紙は「問題番号 + `A.` + 1 本線」
- セットごとに QR コードを右上に出す

このため、図版を伴う問題、グラフを描かせる問題、途中式を多く書かせる問題、表に記入させる問題には、そのままでは対応しにくい。

参照:

- [`../src/lib/print-service.ts`](../src/lib/print-service.ts)
- [`../src/lib/print-document.ts`](../src/lib/print-document.ts)
- [`../src/lib/print-pdf/render-service.ts`](../src/lib/print-pdf/render-service.ts)
- [`../docs/problem-output-logic.md`](./problem-output-logic.md)

### 3.4 採点

現状実装から見ると、採点は英語の短答問題を前提にしている。

- 採点 prompt は「あなたは英語教師として、生徒の解答用紙を採点してください」と定義されている。
- 採点コンテキストは `question`, `answer`, `acceptedAnswers` が中心である。
- 問題の順番に対応する回答を採点する設計であり、複雑な途中式、作図、グラフ、考察記述は明示的に扱っていない。

参照:

- [`../src/lib/grading-service.ts`](../src/lib/grading-service.ts)
- [`../instructions/grading-prompt.md`](../instructions/grading-prompt.md)

### 3.5 現状制約の要点

現状実装から見ると、制約は次の 6 点に要約できる。

1. 問題本文が文字列 1 本である。
2. 管理 UI が生テキスト入力中心である。
3. 印刷テンプレートが短答前提である。
4. セット内問題数が固定である。
5. 外部画像や複雑図版の印刷前提が弱い。
6. 採点 prompt と採点 schema が英語短答前提である。

## 4. 他科目との差分整理

この章で決めること: 英語と比べて、理科・数学では何が増え、どの設計が必要になるかを整理する。

| 観点 | 英語 | 数学 | 理科 | 設計影響 |
| --- | --- | --- | --- | --- |
| 問題本文 | 文字列中心で成立しやすい | 数式、図形、グラフ、座標平面が必要 | 実験図、観察図、表、グラフ、写真が必要 | 本文を構造化ブロックで持つ必要がある |
| 解答形式 | 単語、短文、穴埋めが中心 | 数値、式、途中式、作図、証明、グラフ | 用語、数値、表記入、作図、考察 | 解答欄を問題形式ごとに切り替える必要がある |
| 印刷余白 | 比較的少なくて済む | 問題直下の作業スペースが必要 | 表記入欄、考察欄、グラフ欄が必要 | ページ分割を問題種別で変える必要がある |
| 採点 | 短答の自動採点がしやすい | 最終答は可能だが途中式や証明は難しい | 用語短答は可能だが考察は難しい | 自動採点対象を明確に限定する必要がある |
| アセット依存 | 低い | 中〜高い | 高い | 問題本文とアセット管理を分離する必要がある |

要するに、理科・数学では「問題文」そのものがレイアウトを持つ。したがって、英語と同じ「問題文 = 文字列」のモデルをそのまま拡張するのではなく、「問題文 = 構造化されたドキュメント」として扱う必要がある。

## 5. 単元設計方針

この章で決めること: 理科・数学の単元を、既存の `CoreProblem` を活かしながらどう構造化するかを決める。

### 5.1 基本方針

- 現行の `CoreProblem` は維持する。
- `CoreProblem` を「学習順序制御に使う単元ノード」として引き続き利用する。
- 理科・数学向けには、`CoreProblem` に教科固有のメタデータを追加する前提で設計する。

### 5.2 数学の単元軸

数学は、学年と内容領域を基準に整理する。

- 学年: 中1 / 中2 / 中3
- 領域: `A 数と式 / B 図形 / C 関数 / D データの活用`

例:

- 中1 / A 数と式 / 正の数・負の数
- 中1 / A 数と式 / 文字式
- 中1 / A 数と式 / 一次方程式
- 中1 / C 関数 / 比例・反比例
- 中2 / B 図形 / 合同
- 中3 / C 関数 / 二次関数

### 5.3 理科の単元軸

理科は、学年と分野を基準に整理する。

- 学年: 中1 / 中2 / 中3
- 分野: `第1分野(エネルギー / 物質)`、`第2分野(生命 / 地球)`

例:

- 中1 / 第1分野 / 身の回りの物質
- 中1 / 第2分野 / 植物の生活と種類
- 中2 / 第1分野 / 電流とその利用
- 中2 / 第2分野 / 気象とその変化
- 中3 / 第1分野 / 運動とエネルギー
- 中3 / 第2分野 / 地球と宇宙

### 5.4 `CoreProblem` に追加したいメタデータ

理科・数学を前提にすると、将来的には `CoreProblem` に次のメタデータが必要になる。

- `gradeLevel`
- `domain`
- `curriculumCode`
- `tags`
- `defaultPrintTemplate`
- `defaultGradingMode`

### 5.5 単元コードの考え方

単元コードは将来の seed、import、分析、外部連携を安定させるために持つ。

例:

- 数学: `M1-A-03`
- 理科: `S2-F1-04`

ここでの意味は次の通りとする。

- `M`: 数学
- `S`: 理科
- `1/2/3`: 学年
- `A/B/C/D` または `F1/F2`: 領域・分野
- 末尾番号: 領域内の通し番号

## 6. 想定する問題形式

この章で決めること: 理科・数学で扱う問題形式の一覧と、それぞれの取り扱い方を定義する。

| 形式 | 想定用途 | 必要な本文要素 | 必要な解答欄 | 印刷難易度 | 紙自動採点 |
| --- | --- | --- | --- | --- | --- |
| `SHORT_TEXT` | 用語記述、語句記入 | 段落、空欄 | 短答欄 | 低 | 可 |
| `NUMERIC` | 計算結果、測定値 | 段落、単位表記 | 数値欄 | 低 | 可 |
| `MULTIPLE_CHOICE` | 選択問題、正誤 | 選択肢ブロック | 記号欄 | 低 | 可 |
| `MULTI_BLANK` | 複数空欄補充 | 段落、複数空欄 | 複数短答欄 | 中 | 可 |
| `FORMULA_FINAL` | 最終式のみを答える問題 | 数式、段落 | 式記入欄 | 中 | 条件付きで可 |
| `TABLE_FILL` | 表の穴埋め、実験結果記入 | 表、段落 | 表記入欄 | 中 | 条件付きで可 |
| `DIAGRAM_LABEL` | 図のラベル付け | 画像、SVG、図版注釈 | 図中または別紙記号欄 | 高 | 条件付きで可 |
| `GRAPH_READ` | グラフ読解、値の読み取り | グラフ画像、座標、説明 | 数値欄、記述欄 | 中 | 条件付きで可 |
| `GRAPH_DRAW` | グラフ作成 | 座標平面、補助目盛 | 専用グラフ欄 | 高 | 原則不可 |
| `GEOMETRY_DRAW` | 作図、図形問題 | SVG 図形、条件文 | 作図欄 | 高 | 原則不可 |
| `EXPERIMENT_RECORD` | 実験記録、観察整理 | 表、図、写真、段落 | 表欄、記述欄 | 高 | 原則不可 |
| `SHORT_EXPLANATION` | 理由説明、考察、証明要点 | 段落、図、表 | 記述欄 | 高 | 条件付きまたは手動 |

### 6.1 問題形式の扱い方

- `SHORT_TEXT`, `NUMERIC`, `MULTIPLE_CHOICE`, `MULTI_BLANK` は、既存の印刷・採点構造に比較的近い。
- `FORMULA_FINAL`, `TABLE_FILL`, `DIAGRAM_LABEL`, `GRAPH_READ` は、構造化ブロックと専用の印刷テンプレートが必要である。
- `GRAPH_DRAW`, `GEOMETRY_DRAW`, `EXPERIMENT_RECORD`, `SHORT_EXPLANATION` は、印刷はできても、紙自動採点を既定にすべきではない。

## 7. 問題管理仕様

この章で決めること: 問題本文、アセット、解答、印刷設定、採点設定をどう分けて持つかを定義する。

### 7.1 基本方針

- 生テキスト管理を主仕様にしない。
- 問題本文は構造化ブロックで管理する。
- 問題本文とアセットは分離する。
- 問題ごとに印刷設定と採点設定を持つ。
- 既存の `question` は後方互換用のプレーンテキスト要約または fallback として残す前提にする。

### 7.2 問題データの概念構造

概念上、理科・数学向けの問題は次のような情報を持つ。

```ts
type ProblemSpec = {
  problemType: ProblemType;
  formatVersion: number;
  questionSummary: string;
  bodyBlocks: BodyBlock[];
  answerSpec: AnswerSpec;
  printConfig: PrintConfig;
  gradingConfig: GradingConfig;
  assets: ProblemAssetRef[];
};
```

### 7.3 `bodyBlocks` の考え方

本文ブロックは、リッチテキストを自由入力させるのではなく、用途が明確なブロックの組み合わせで定義する。

想定するブロック例:

- `paragraph`
- `mathInline`
- `mathDisplay`
- `image`
- `svgFigure`
- `table`
- `choices`
- `blankGroup`
- `graphGrid`
- `coordinatePlane`
- `caption`

この方式を採る理由は次の通りである。

- 印刷レイアウトの制御がしやすい。
- PDF 生成時の崩れを抑えやすい。
- 採点用にどの要素が問題本文かを抽出しやすい。
- 管理画面で入力コンポーネントを分けやすい。

### 7.4 `answerSpec` の考え方

解答仕様は本文とは別に管理する。

含めたい情報の例:

- 解答形式
- 解答欄の数
- 許容単位
- 許容誤差
- 複数正答
- 解答順序の厳密さ
- 手動採点必須かどうか

### 7.5 `printConfig` の考え方

問題ごとに、どのテンプレートで印刷するべきかを持たせる。

例:

- `template = compact`
- `template = workspace`
- `answerAreaMode = inline`
- `estimatedHeight = medium`
- `avoidBreakInside = true`

### 7.6 `gradingConfig` の考え方

問題ごとに採点方式を定義する。

例:

- `gradingMode = exact_match`
- `gradingMode = numeric_tolerance`
- `gradingMode = multi_blank`
- `gradingMode = rubric_ai`
- `gradingMode = manual_review`

### 7.7 アセット管理

問題本文に画像や図版を直接埋め込むのではなく、アセットは分離して管理する。

想定する項目:

- `kind`
- `storageKey`
- `mimeType`
- `width`
- `height`
- `altText`
- `source`
- `checksum`

本文側はアセット参照のみを持つ。

なお、ここで分離したいのは「DB」と「バイナリアセット」の責務である。

- 問題のメタデータ、本文構造、採点設定、単元紐付けは Supabase DB に保持する。
- 画像、SVG、GeoGebra 由来の書き出し画像、Desmos の書き出し画像などのバイナリアセットは Supabase Storage に保持する。

つまり、問題の主データを Supabase に寄せること自体は妥当であり、Google Drive を避けたい理由は DB の置き場ではなく、教材アセットの配信元としての性質にある。

### 7.8 一括登録方針

TSV / スプレッドシート一括登録は残すが、対象は単純問題だけに限定する。

一括登録の対象候補:

- `SHORT_TEXT`
- `NUMERIC`
- `MULTIPLE_CHOICE`
- `MULTI_BLANK`
- 一部の `FORMULA_FINAL`

一括登録の対象外候補:

- 画像必須の問題
- SVG 図形が必要な問題
- 表レイアウトが複雑な問題
- グラフ作成問題
- 作図問題

複雑問題は、構造化 UI で作成する前提にする。

## 8. 印刷仕様

この章で決めること: 理科・数学の問題を、今のようにきれいに印刷するための基本仕様を定義する。

### 8.1 基本方針

- 画面プレビューと PDF は同じレンダラを使う。
- テキストだけでなく、数式、SVG、表、画像を同一レンダリングパイプラインで扱う。
- 1セット 10 問固定は廃止し、問題の高さとテンプレートに応じてページを分割する。
- 問題形式ごとに印刷テンプレートを切り替える。

### 8.2 表現技術の選定

理科・数学では、問題本文の表現技術を 1 つに寄せるのではなく、役割ごとに分ける。

| 技術 | 向いている用途 | 推奨度 | 位置づけ |
| --- | --- | --- | --- |
| LaTeX | 数式のソース表現 | 高 | 数式の canonical source |
| KaTeX | LaTeX の画面表示・印刷表示 | 高 | Web/PDF の数式レンダラ |
| SVG | 図形、座標平面、回路図、模式図 | 高 | 印刷用図版の canonical format |
| Desmos | グラフの作成、教師向けインタラクティブ確認 | 中 | authoring / preview 用 |
| GeoGebra | 図形、関数、幾何、動的教材 | 中 | authoring / preview 用 |
| MathML | 数式の標準表現 | 中 | 補助的な互換・アクセシビリティ用途 |

#### LaTeX / KaTeX

数式のソースは LaTeX で保持し、表示と印刷は KaTeX を第一候補とする。

理由:

- LaTeX は数学表現の入力形式として実績がある。
- KaTeX はサーバー側で `renderToString` により HTML 化できる。
- KaTeX は `htmlAndMathml` 出力を使えるため、印刷と可読性を両立しやすい。

#### SVG

SVG は、理科・数学の印刷図版の主形式として最も有力である。

理由:

- ベクター形式のため印刷時に劣化しにくい。
- DOM と親和性が高く、HTML/PDF レンダリングに組み込みやすい。
- 線、ラベル、座標、図形の制御がしやすい。

図形、回路図、座標平面、模式図、ラベル付き図版は、可能な限り SVG を canonical format にする。

#### Desmos

Desmos は、グラフ問題の authoring と interactive preview には有力である。

ただし、Desmos の本番利用には API key が必要であり、埋め込みも JavaScript ベースになる。そのため、印刷の canonical source にするよりは、次の用途に限定するのが妥当である。

- 教師がグラフを作る
- 生徒向けにインタラクティブ確認画面を出す
- 出来上がったグラフを画像として書き出し、印刷用アセットに変換する

つまり、Desmos は「問題の本体」ではなく「グラフ作成・確認ツール」として扱う。

#### GeoGebra

GeoGebra は、図形、幾何、関数、動的教材の authoring と interactive preview に向いている。

ただし、GeoGebra も JavaScript 埋め込みまたは iframe 埋め込みが中心であり、PDF 印刷の canonical source として直接依存すると不安定要素が増える。そのため、次の用途を推奨する。

- 作図・幾何教材の作成
- 教師向け・生徒向けの動的プレビュー
- `.ggb` や `material_id` を補助情報として保持
- 印刷時は SVG または画像へ書き出した静的アセットを使う

つまり、GeoGebra も「問題の本体」ではなく「作図・教材作成ツール」として扱う。

#### MathML

MathML は標準技術として有望だが、v1 の主手段にはしない。

理由:

- ブラウザ実装は改善しているが、印刷再現性を最優先にすると KaTeX のほうが制御しやすい。
- v1 では入力形式を LaTeX に寄せたほうが、問題作成フローを整えやすい。

### 8.3 図版・画像方針

- 図形、座標、回路図、模式図は SVG を優先する。
- 写真や顕微鏡画像などは raster 画像を許容する。
- 画像は印刷を前提に、十分な解像度を確保する。
- 画像読込完了を待ってから PDF を生成する。
- Desmos と GeoGebra を使う場合も、印刷で使う最終成果物は SVG または raster 画像として保持する。

### 8.4 印刷テンプレート

| テンプレート | 主用途 | 特徴 |
| --- | --- | --- |
| `compact` | 短答、選択、数値 | 問題数を多く載せる。余白は最小限 |
| `standard` | 通常問題 | 問題本文と標準的な解答余白を持つ |
| `workspace` | 計算、途中式、考察 | 問題直下に広めの余白を確保する |
| `graph` | グラフ、作図 | 専用の座標欄や作図欄を持つ |

### 8.5 解答欄モード

| モード | 用途 | 仕様 |
| --- | --- | --- |
| `別紙短答` | 短答、記号、数値 | 現行に近い別紙解答用紙 |
| `問題直下余白` | 計算、途中式、考察 | 問題の直下に作業欄を置く |
| `グラフ専用ページ` | グラフ作成、作図 | 問題とは別に専用ページを付ける |
| `手動確認用` | 長文記述、レポート | 採点用ではなく提出用に近い紙面にする |

### 8.6 改ページルール

- 図版と本文は原則として同一ページに収める。
- `avoidBreakInside` を持つ問題は途中改ページを避ける。
- `workspace` と `graph` は、問題ごとの高さが大きいため、ページ埋めではなく問題単位での改ページを優先する。
- 問題が複数ページに跨る場合は、図版を前ページに残して本文だけ次ページに送らない。

### 8.7 QR コード

QR の基本役割は、現行どおり「生徒 ID と問題 ID 群を採点系へ渡すこと」とする。

現時点では、v1 の QR ペイロードは次を維持する前提とする。

- 生徒識別子
- 問題 ID 群
- 単元トークン

ただし、将来的に複雑な回答エリアを扱う場合は、追加メタデータを載せる余地を残す。

## 9. 採点への影響

この章で決めること: 理科・数学を入れたときに、自動採点の対象範囲をどこまでにするかを定義する。

### 9.1 基本方針

- 自動採点は問題形式ごとに適用範囲を制限する。
- 全問題を紙自動採点対象にはしない。
- 人手確認が必要な問題形式を、仕様上正式に認める。

### 9.2 採点モード

| 採点モード | 主な対象 | v1 方針 |
| --- | --- | --- |
| `exact_match` | 用語、記号、単純短答 | 自動採点対象 |
| `numeric_tolerance` | 数値解答、単位付き数値 | 自動採点対象 |
| `choice` | 選択問題 | 自動採点対象 |
| `multi_blank` | 複数空欄 | 自動採点対象 |
| `formula_final` | 最終式のみ | 条件付きで自動採点対象 |
| `rubric_ai` | 短い理由説明、簡単な考察 | 補助的に利用 |
| `manual_review` | 作図、証明、長文考察 | 手動採点対象 |

### 9.3 自動採点対象の考え方

v1 で自動採点対象にすべきなのは、次である。

- 短答
- 数値
- 記号選択
- 複数空欄
- 最終答のみを答える計算問題

v1 で自動採点対象にすべきでないのは、次である。

- 途中式の妥当性
- 証明の論理構成
- グラフ作成の妥当性
- 作図の正確性
- 長文考察

### 9.4 英語 prompt の流用不可

現状実装から見ると、採点 prompt は英語短答向けである。そのため、理科・数学では次が必要になる。

- 科目別 prompt
- 問題形式別 prompt
- 採点モード別 schema
- `ProblemForGrading` の拡張

つまり、英語の採点設計を流用するのではなく、「教科別 + 問題形式別」の採点設計に切り替える必要がある。

## 10. 技術的懸念点

この章で決めること: 実装前に認識しておくべき技術リスクを明示する。

| 懸念点 | 現状実装から見える状況 | 影響 | 仕様上の対策 |
| --- | --- | --- | --- |
| 外部画像の PDF 読込待ち不足 | PDF 生成時にフォント待ちはあるが、画像読込完了待ちが明示されていない | 印刷時に画像欠落の可能性がある | 画像読込完了を待ってから `page.pdf()` する |
| KaTeX / MathML の印刷再現性 | 数式レンダリング基盤が未導入 | 数式崩れやフォント差異が発生しうる | v1 は KaTeX を主手段に固定する |
| 画像ストレージの選定 | Drive は答案取込には使っているが、教材アセット配信には不向き | URL 管理、配信安定性、権限制御が煩雑になる | 教材アセットは専用ストレージで管理する |
| 問題タイプ混在時のページ崩れ | 現状はテキスト主体の単純レイアウト | 余白不足、図版の分断、解答欄不足が起こる | テンプレート型印刷に切り替える |
| 既存英語問題との後方互換 | `question` テキスト前提のロジックが多い | 既存データが壊れる可能性がある | `question` を fallback として残す |
| 採点対象外問題の UI 扱い | 現状は全問題が同じように扱われる | 自動採点されない問題で誤解が生じる | `manual_review` を UI 上で明示する |

### 10.1 Google Drive の論点は DB ではなくアセット配信である

問題の主データを Supabase DB に保持する前提であれば、確かに「Google Drive が DB として問題になる」という話ではない。

この仕様で懸念しているのは、Google Drive を教材アセットの配信元や実行時依存先にすることである。

つまり、論点は次の切り分けになる。

- Supabase DB: 問題メタデータや本文構造を保持する場所
- Supabase Storage: 画像や SVG などの教材アセットを配信する場所
- Google Drive: 現状では答案の受け取りや運用上のファイル連携で使っている場所

Google Drive は、答案の受け取りや人間向けのファイル共有には向いているが、教材アセットの恒常配信には向きにくい。

主な理由:

- 署名 URL や公開制御の前提が教材配信向きではない
- PDF 生成時に安定した画像取得を担保しにくい
- キャッシュ戦略を組みにくい
- 問題アセットとしてのメタデータ管理がしにくい

したがって、教材アセットは Supabase DB とは別に、Supabase Storage などの専用ストレージで管理する前提で考えるべきである。

## 11. 推奨方針

この章で決めること: 本書としての推奨案を 1 つに絞って示す。

本書の推奨方針は次の通りとする。

1. 問題管理は、生テキストではなく構造化入力を主軸にする。
2. 単元ノードは現行の `CoreProblem` を維持し、必要なメタデータを追加する。
3. 問題本文とアセットは分離して管理する。
4. 印刷は問題形式ごとのテンプレート型にする。
5. 数式は KaTeX、図版は SVG 優先で扱う。
6. 紙自動採点の対象は限定し、`manual_review` を正式に認める。
7. 既存英語問題は後方互換を保ちつつ、理科・数学のみ新仕様へ段階移行する。

要するに、「構造化入力主軸」「`CoreProblem` 維持」「タイプ別印刷」「自動採点範囲を限定」が、Sullivan にとって最も破綻しにくい拡張方針である。
