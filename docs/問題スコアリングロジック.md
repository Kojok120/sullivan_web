# 問題スコアリングロジック

プリント問題選出に使われるスコアリングの「考え方」と「具体的な挙動」を、定数値ではなく言葉と例で説明するドキュメント。

実装上の式・定数の正確なリファレンスは [`problem-output-logic.md`](./problem-output-logic.md) §3 を参照。

実装: `src/lib/print-algo.ts`

---

## 1. 基本の考え方

プリントに出す問題を選ぶときに、3つの観点でスコアを付ける。

1. **未着手の問題は最優先で出したい** — まだ一度も解いていない問題は学習機会を与えるべき
2. **時間が経った問題は復習すべき** — 忘却曲線：解いた直後より、しばらく経った問題の方が復習価値が高い
3. **間違えた問題は弱点なので多めに、正解した問題は控えめに**

この3つを1つのスコアにまとめて、上位から印刷する。

---

## 2. 未着手の問題

一度も解いていない問題には、**他の何より高い固定スコア**を与える。

これによって、長期間放置された既着手問題に押し出されず、必ず未着手問題が印刷の上位に並ぶ。問題の `order`（教材内の並び順）でほんの少しだけ前のものを優先する程度の差はあるが、基本的には未着手問題はスコアでは横一線で、シャッフルによってランダムに選ばれる。

---

## 3. 既着手の問題（忘却曲線）

最後に解いた日からの **経過日数に比例してスコアが上がる**。

ただし上限がある。**80日以上放置された問題は、それ以上放置してもスコアは上がらない**。これは「100日前の問題と500日前の問題に優先度の差を付ける意味は薄い」という割り切り。これがないと、何ヶ月も触っていない古い問題が永遠に最上位に居座り続けて、復習リストが化石化する。

経過日数だけで決まる素のスコアは、おおよそこんなイメージ。

| 経過日数 | 素の忘却スコア |
|---|---|
| 1日 | 10 |
| 7日 | 70 |
| 14日 | 140 |
| 30日 | 300 |
| 80日以上 | 800（頭打ち） |

---

## 4. 正解 / 不正解の補正

上の忘却スコアに、**正解か不正解かで加点 / 減点**する。

- **不正解だった問題** — 弱点として **+100 のボーナス** を加算
- **正解した問題** — 習得済みなので **-150 のペナルティ** を減算

この差（合計250ポイント）は、不正解問題が経過日数25日分くらい先行する効果がある。つまり感覚的には、

> 「今日不正解だった問題」は「3週間前に正解した問題」とほぼ同じ復習優先度になる

くらいの目盛り感。

---

## 5. スコア早見表

| 状態 | スコア計算 | 想定スコア |
|---|---|---|
| 未着手 | 固定の高スコア | **約1500** |
| 不正解・1日経過 | 10 + 100 | 110 |
| 不正解・14日経過 | 140 + 100 | 240 |
| 不正解・80日以上 | 800 + 100 | **900**（既着手の最大） |
| 正解・1日経過 | 10 − 150 | −140 |
| 正解・14日経過 | 140 − 150 | −10 |
| 正解・80日以上 | 800 − 150 | **650**（正解の最大） |

スコア優先順序は **未着手 (1500) > 不正解 (≤900) > 正解 (≤650)** で、構造的にこの順序が崩れない。

---

## 6. スロット分割（重要）

スコア順に上位から取るだけだと、未着手が多いと復習が消え、未着手が少ないと既着手だけで埋まってしまう。これを防ぐため、印刷枠を **未着手枠と既着手枠に分けて確保**する。

- まず未着手枠を確保（後述の `newQuota`）
- 次に既着手枠を確保（不正解→正解の順に並ぶ）
- 既着手が枯渇したら、未着手の続きで残りを埋める（オーバーフロー）

---

## 7. 出題数による違い

スロット分割の比率は出題数で変わる。

### count = 30（既定値）

未着手枠は出題数の **40%** を確保する。よって **未着手12 / 既着手18**。

### count = 10（運用値）

40% だと4問になるが、未着手枠の **最低保証5問** が効いて **未着手5 / 既着手5** になる。

| 状況 | 未着手枠 | 既着手枠 | overflow | 出力 |
|---|---|---|---|---|
| 未着手 ≥5 / 既着手 ≥5 | 5 | 5 | 0 | 未着手5 + 既着手5 |
| 未着手 100 / 既着手 0 | 5 | 0 | 5 | 未着手10 |
| 未着手 0 / 既着手 100 | 0 | 10 | 0 | 既着手10 |
| 未着手 3 / 既着手 100 | 3 | 7 | 0 | 未着手3 + 既着手7 |

count=10 だと未着手が常に半分を占めるので、未着手プールが豊富な生徒では復習が圧迫されやすい。
ただし、現状の運用方針として「未着手を出さないと前に進まない」課題を解消するためにこの比率を採用している。

---

## 8. 同点処理

同一スコアの問題群（典型的には未着手の同率1500点グループ）は、seed 固定でシャッフルする。

- 同じプレビューを再表示しても順序は変わらない
- プレビューを作り直したときのみ再抽選される
- seed = 明示指定 or `${userId}:${subjectId}:${coreProblemId ?? 'all'}`

これにより、「印刷ボタンを2回押したら別の問題が出てきた」という混乱を防ぐ。

## 9. 具体例

### 例1: スコア比較

| 問題 | 状態 | 経過日数 | スコア |
|---|---|---|---|
| A | 未着手 | − | **1500** |
| B | 不正解 | 100日 | 800 + 100 = **900**（上限） |
| C | 不正解 | 14日 | 140 + 100 = **240** |
| D | 不正解 | 1日 | 10 + 100 = **110** |
| E | 正解 | 100日 | 800 − 150 = **650**（上限） |
| F | 正解 | 14日 | 140 − 150 = **−10** |
| G | 正解 | 1日 | 10 − 150 = **−140** |

並べ替え: **A(1500) > B(900) > E(650) > C(240) > D(110) > F(−10) > G(−140)**

ポイント:
- A は何があっても最上位（未着手の壁は高い）
- B（100日不正解）と E（100日正解）の差は 250 → 正解はまだ覚えてる可能性が高いから後回し
- C（2週間前の不正解）よりも E（100日前の正解）が上 → 長期間触っていない正解問題の方が復習価値が高いという判断
- F・G（直近に正解）はマイナス → 解いたばかりで明らかに覚えているので、ほぼ印刷されない

### 例2: すぐ復習の挙動

- 昨日不正解 (D = 110) vs 2週間前正解 (F = −10) → 昨日不正解の方が圧倒的に上位
  - 「間違えた問題は記憶が新しいうちに戻ってくる」という直感に合う
- 2ヶ月前正解 (600 − 150 = 450) vs 1週間前不正解 (70 + 100 = 170) → 2ヶ月前正解が上
  - 「正解しても2ヶ月触らなければ復習対象に戻ってくる」設計

### 例3: 10問印刷の挙動（標準）

ある生徒の状態:
- 未着手: 50問
- 既着手・不正解: 8問（全部1週間以内）
- 既着手・正解: 200問（バラバラ）

→ **未着手5 + 既着手5**
- 未着手枠: 上位5問（同率1500、シャッフルで決まる）
- 既着手枠: 不正解上位5問（不正解8問のうちスコア上位5問）
- 正解問題は今回1問も印刷されない

### 例4: 未着手が枯れた場合（10問）

- 未着手: 3問
- 既着手・不正解: 0問
- 既着手・正解: 100問

→ **未着手3 + 既着手7**（オーバーフロー無し）
- 既着手側は正解しか無いので、長期放置の正解問題上位7問が選ばれる

### 例5: 入会直後（10問）

- 未着手: 大量
- 既着手: 5問だけ

→ **未着手5 + 既着手5**（既着手は全件出る）
- もし既着手が3問しかなければ「未着手5 + 既着手3 + overflow 2 = 未着手7 + 既着手3」になる

## 10. このロジックが見ているもの / 見ていないもの

### 見ているもの
- `Problem.coreProblems` — 候補絞り込みに使用
- `Problem.order` — 同順位での僅差優先
- `UserProblemState.lastAnsweredAt` — 未着手判定 + 経過日数
- `UserProblemState.isCleared` — 正解 / 不正解の分岐
- `UserCoreProblemState.isUnlocked` / `isLectureWatched` — おまかせ印刷時の ready set 判定

### 見ていないもの（混同注意）
- `UserProblemState.priority` — 採点時に A:−10 / B:−5 / C:+5 / D:+10 で更新されるが、印刷ロジックは読まない
- `UserProblemState.unlockLastAnsweredAt` / `unlockIsCleared` — アンロック判定専用
- 学年・難易度・タグ等

## 11. 関連ロジックとの違い

「印刷スコアリング」と「アンロック判定」は別物なので混同しないこと。

| | 印刷スコアリング | アンロック判定 |
|---|---|---|
| 実装 | `src/lib/print-algo.ts` | `src/lib/progression.ts` / `grading-service.ts` |
| 使うフィールド | `lastAnsweredAt` / `isCleared` | `unlockLastAnsweredAt` / `unlockIsCleared` |
| 単元別印刷の回答を反映するか | する | **しない**（isUnitMode は unlock\* を更新しない） |
| 役割 | 次に出す10問を決める | 次の CoreProblem を解放するか決める |
