## 1. あなたがやるべき設定作業 (Action Items)

### A. Supabase (Database)
1.  Supabase ダッシュボードの画面上部にある **"Connect"** ボタンをクリックします。
2.  **"Connection String"** タブ -> **"URI"** を選択します。
3.  **"Mode"** (または "Pool Mode") の切り替えスイッチでポートを確認します。
    *   **Transaction Mode (Port 6543)**: このURIを `DATABASE_URL` (末尾に `?pgbouncer=true` を付与) として使用します。
    *   **Session Mode (Port 5432)**: このURIを `DIRECT_URL` として使用します。

### B. Upstash (QStash & Redis)
1.  Upstash Console で **Redis Database** を作成。
    *   **REST URL** と **REST Token** を取得。
2.  **QStash** タブを開く。
    *   `QSTASH_TOKEN` と `QSTASH_CURRENT_SIGNING_KEY` を取得。

### C. Google Cloud Project & Secret Manager (重要: セキュリティ設定)
サービスアカウントキー (`service-account.json`) を安全に管理するための推奨手順です。

1.  **Secret Manager の有効化**:
    Google Cloud Console で「Secret Manager」を検索し、APIを有効にします。
2.  **シークレットの作成**:
    *   "CREATE SECRET" をクリック。
    *   Name: `sullivan-service-account` (例)
    *   Secret value: ローカルにある `service-account.json` の中身を丸ごとコピー＆ペーストします。
3.  **Cloud Run サービスアカウントへのアクセス権付与**:
    *   Cloud Run が実行されるデフォルトのサービスアカウント（通常 `[project-number]-compute@developer.gserviceaccount.com`）に対し、このシークレットへの「**Secret Manager Secret Accessor**」ロールを付与します。
    *   (IAM & Admin -> IAM ではなく、Secret Manager の Permission タブから追加するのが簡単です)

### 3. トラブルシューティング: DBマイグレーションが止まる場合
もし `npx prisma migrate deploy` が応答しなくなる場合は、接続プールの問題の可能性があります。以下のコマンドで直接接続 (`DIRECT_URL`) を強制的に使用して実行してください。

```bash
export $(grep DIRECT_URL .env | xargs) && DATABASE_URL=$DIRECT_URL npx prisma migrate deploy
# 続けてシード実行
export $(grep DIRECT_URL .env | xargs) && DATABASE_URL=$DIRECT_URL npx prisma db seed
```

### トラブルシューティング実績 (今回の対応)
今回のデプロイメントで以下の問題を解決しました：
1.  **IAM権限不足**: デフォルトのサービスアカウントにビルド権限 (`roles/cloudbuild.builds.builder`) とシークレットアクセス権限 (`roles/secretmanager.secretAccessor`) を付与しました。
2.  **ビルドエラー (Prisma)**: `package.json` に `postinstall: prisma generate` を追加し、Dockerfileで `deps` ステージに `prisma` フォルダをコピーすることで解消しました。
3.  **環境変数依存**: `auth.ts` の `process.env.JWT_SECRET` チェックを遅延評価に変更し、ビルド時に環境変数がなくても失敗しないように修正しました。
4.  **デプロイ設定漏れ**: `deploy.sh` に `JWT_SECRET` が含まれていなかったため追加しました。
5.  **DBスキーマ不整合**: `LearningHistory` テーブルに `groupId` カラム等が不足しておりログイン時にエラーが発生していましたが、手動マイグレーションで修正しました。
6.  **問題登録の競合解消**: 削除したはずの問題が「ゾンビ」として残り再登録を妨害していましたが、手動削除および「Cascade Delete」設定の適用により、削除が正しく機能するように修正しました。
7.  **AI採点の自動化**: Cloud Runの特性上、バックグラウンドでのポーリング停止してしまうため、**Cloud Scheduler** を設定して1分ごとに採点トリガーを実行するようにしました。
8.  **権限設定の修正**: Cloud Run上でのGoogle Drive連携認証において、認証キーの渡し方が誤っておりクラッシュしていた問題を修正しました（`/app` へのマウントによるコード隠蔽を回避するため、`/secrets` ディレクトリへマウント先を変更）。
    *   *補足*: デプロイ時に「環境変数の型競合エラー」が発生したため、一度 `gcloud run services update` で設定をクリアしてから再デプロイを行いました。
9.  **ファイルシステム仕様対応**: Cloud Runのファイルシステムは読み取り専用であるため、一時ファイルの保存先を `/app/tmp` から システム一時領域（`/tmp`）に変更しました。
10. **OpenCV依存関係の追加**: Dockerfileに `libgl1` と `libglib2.0-0` を追加し、Cloud Run環境でのQRコード読み取りエラー（ImportError）を解消しました。
    *   *運用注意*: 処理に失敗したファイルは `[ERROR]` 接頭辞がつき、以降の自動処理から**除外**されます。再試行する場合は、ファイルを再アップロードするか、名前を変更してください。
11. **メモリ/CPU設定の最適化 (重要)**: ログ調査により、OpenCVとAI処理のメモリ不足（OOM）によるクラッシュが判明したため、Cloud Runのリソース割り当てを増強しました。
    *   `deploy.sh` に以下を追加: `--memory 4Gi` と `--cpu 2`
    *   *結果*: ログ消失問題が解消し、エンドツーエンドの採点処理が完了することを確認しました。
12. **Geminiプロンプトのバグ修正**: `grading-service.ts` のGemini QRスキャン用プロンプトに**ハードコードされた例示ID**が含まれており、Geminiが毎回その例示IDを返していた問題を修正しました。
13. **OpenCV 4.6→4.12アップグレード**: Debian apt の `python3-opencv` (4.6.0) には `QRCodeDetectorAruco` が含まれていないため、Dockerfileを変更してpip `opencv-python-headless` (4.12+) をインストールするように変更しました。
    *   *理由*: 600dpi (6842x4816px) の大きなスキャン画像では、標準の `QRCodeDetector` が失敗します。`QRCodeDetectorAruco` は4.8+で追加された高性能な検出器で、ローカルテストで600dpi画像のQRコード読み取りに成功しました。
14. **QRリーダーの強化**: `scripts/qr_reader.py` に7段階の検出メソッドを追加しました：
    1. 標準検出 (Original Image)
    2. グレースケール変換
    3. 適応的閾値処理
    4. 拡大処理 (小画像用)
    5. Aruco検出器 (OpenCV 4.8+)
    6. コーナークロップ (4隅をそれぞれ検索)
    7. ダウンスケール (巨大画像用)

*   **Schedule**: 毎分 (`* * * * *`)
*   **Auth**: Unauthenticated (Service Account configuration available if needed)

## 6. トラブルシューティング

### Q. Driveのファイルが `[ERROR]` になる
*   **原因 1**: **データベースに存在しない生徒/問題ID用のQRコードを使用している**。
    *   開発環境(Local)で発行したQRコードは、本番環境(Cloud Run/Supabase)では無効（ID不一致）な場合があります。
    *   **対策**: 本番アプリ上で生徒を作成し、新しいQRコードを発行・印刷して使用してください。
*   **原因 2**: AI採点処理中のエラー。
    *   Cloud Runのログを確認し、具体的なエラーメッセージ（例: `PrismaClientKnownRequestError`）を特定する必要があります。

### Q. ログインできない
*   `JWT_SECRET` が設定されているか、Supabaseのユーザーが存在するか確認してください。

シークレットをボリュームとしてマウントし、環境変数 `GOOGLE_APPLICATION_CREDENTIALS` でそのパスを指定します。

```bash
# 1. Google Cloud 認証
gcloud auth login
gcloud config set project [YOUR_PROJECT_ID]

# 2. Cloud Run へデプロイ
# --update-secrets でシークレットを /secrets/google-creds にマウントします
gcloud run deploy sullivan-app \
  --source . \
  --platform managed \
  --region asia-northeast1 \
  --allow-unauthenticated \
  --set-env-vars "NODE_ENV=production" \
  --set-env-vars "DATABASE_URL=postgres://..." \
  --set-env-vars "DIRECT_URL=postgres://..." \
  --set-env-vars "QSTASH_TOKEN=..." \
  --set-env-vars "QSTASH_CURRENT_SIGNING_KEY=..." \
  --set-env-vars "UPSTASH_REDIS_REST_URL=..." \
  --set-env-vars "UPSTASH_REDIS_REST_TOKEN=..." \
  --set-env-vars "APP_URL=https://[YOUR-CLOUD-RUN-URL]" \
  --set-env-vars "GOOGLE_APPLICATION_CREDENTIALS=/secrets/google-creds" \
  --update-secrets=/secrets/google-creds=sullivan-service-account:latest
```
